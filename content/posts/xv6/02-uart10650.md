# 给自己加点料——UART10650驱动重写与移植

我准备使用rust重写UART10650的驱动。

##初始化

首先驱动当然是 `no_std`的：

```rust
#![no_std]
```

我们注意到uart.c里写了一系列常量，宏之类的辅助，我们将其迁移至rust

```rust

pub type RegOffset = usize;
pub type RegValue = u8;

// 原 uart.c 中的 UART 寄存器偏移和位定义。
pub const UART0:usize = 0x10000000;
pub const RHR: RegOffset = 0;
pub const THR: RegOffset = 0;
pub const IER: RegOffset = 1;
pub const IER_RX_ENABLE: RegValue = 1 << 0;
pub const IER_TX_ENABLE: RegValue = 1 << 1;
pub const FCR: RegOffset = 2;
pub const FCR_FIFO_ENABLE: RegValue = 1 << 0;
pub const FCR_FIFO_CLEAR: RegValue = 3 << 1;
pub const ISR: RegOffset = 2;
pub const LCR: RegOffset = 3;
pub const LCR_EIGHT_BITS: RegValue = 3 << 0;
pub const LCR_BAUD_LATCH: RegValue = 1 << 7;
pub const LSR: RegOffset = 5;
pub const LSR_RX_READY: RegValue = 1 << 0;
pub const LSR_TX_IDLE: RegValue = 1 << 5;

pub const UART_TX_BUF_SIZE: usize = 32;

#[inline]
 const fn reg(offset: RegOffset) -> *mut u8 {
    (UART0 + offset) as *mut u8
}

#[inline]
 unsafe fn read_reg(offset: RegOffset) -> RegValue {
    unsafe { core::ptr::read_volatile(reg(offset).cast_const()) }
}

#[inline]
 unsafe fn write_reg(offset: RegOffset, value: RegValue) {
    unsafe { core::ptr::write_volatile(reg(offset), value) };
}

```

其中，我们使用RegOffset和RegValue两个newtype来增强安全性(编译器会检查传给三个inline函数的参数类型)

这里仅仅标注 `#[inline]`而不强制 `#[inline(always)]`,是因为之前在做risc-v模拟器时，出现了强制内联小函数性能回退的问题....大概与icahce miss有关。考虑这里我也懒得跑PGO,仅仅提示编译器进行内联。

## uart驱动分解

首先，驱动提供了一个uartinit()函数来进行一系列初始化。
接下来，面向操作系统，驱动提供了uartputc uartputc_sync  为上层提供写入功能
面向硬件，驱动提供了uartstart 将缓冲区中的数据写入UART
uartgetc 尝试读取
uartintr 处理中断. 在这里我们遇到的中断只有两种可能:

1. 来输入了
2. Uart输出完成，可以继续输出了

于是uartintr处理中断:

1. 尽可能循环调用uartgetc，直到没有数据可读，然后将数据发给console
2. 调用uartstart

所以工作大致是这样的:
uartinit初始化整个模块
uartputc_sync 不经过这一系列机制，直接将数据写入UART(如果已满，则等待)
uartputc 将数据写入缓冲区，然后调用uartstart尝试写入。如果写不完久等下一次uartstart被调用(硬件发中断表示“可以继续写了”)

这是我们“主动”能做的。
被动做的就是硬件发中断，通知我们可以继续发消息/可以接收消息了。
我们就在uartintr尝试接受消息，然后继续调用uartstart发送消息

对外的接口只有uartputc，uartputc_sync，uartintr(在trap处被调用)
需要的外部接口是consoleintr

需要说明，uartgetc仅仅在uartintr被调用...
所以为什么写在defs.h了啊!
删了，似乎没影响，交了个小pr，似乎是unused declearation.
https://github.com/mit-pdos/xv6-riscv/pull/470

## 设计UART模块

我们可以使用面向对象的程序设计方法。根据C程序，我们知道，在这里我们需要
1.维护一个锁
2.维护一段环形缓冲区

那么数据结构可以设计为

```rust
struct UartPointer(usize);
struct UartBuffer{
    buf: [u8; UART_TX_BUF_SIZE],
    uart_writer: UartPointer,
    uart_reader: UartPointer,
}
struct Uart{
    buffer: Spinlock<UartBuffer> //spinlock是之前实现的，简单引入
}
```

UartPointer是一个newtype:

```rust
#[derive(Copy, Clone, Eq, PartialEq)]
  struct UartPointer(usize);

impl UartPointer {
    fn new() -> Self {
        Self(0)
    }
#[inline]
    fn wrapping_add(self, offset: usize) -> Self {
        Self((self.0 + offset) % UART_TX_BUF_SIZE)
    }
#[inline]
    fn wrapping_sub(self, offset: usize) -> Self {
        Self((self.0 + UART_TX_BUF_SIZE - offset % UART_TX_BUF_SIZE) % UART_TX_BUF_SIZE)
    }
}
```

并实现 `From<UartPointer> for usize`,`Add<usize>`,`AddAssign<usize>`,`Sub<usize>`,`SubAssign<usize>`来进一步封装环形缓冲区的操作。

环形缓冲区本身设计push和pop两个操作:

```rust

struct UartBuffer {
    buf: [u8; UART_TX_BUF_SIZE],
    uart_writer: UartPointer,
    uart_reader: UartPointer,
}

impl UartBuffer {
    fn new() -> Self {
        Self {
            buf: [0; UART_TX_BUF_SIZE],
            uart_writer: UartPointer::new(),
            uart_reader: UartPointer::new(),
        }
    }
    //返回输入值：缓冲区已满
    //返回Ok：成功写入
    fn push(&mut self, value: u8)->Result<(),u8>{
        if self.uart_writer == self.uart_reader - 1 {
            // 满了，环给你
            Err(value)
        }
        else{
        self.buf[usize::from(self.uart_writer)] = value;
        self.uart_writer += 1;
        Ok(())
        }
    }
    //返回None：缓冲区已空
    //返回Some：成功读取
    fn pop(&mut self) -> Option<u8> {
        if self.uart_writer == self.uart_reader {
            // 空
            None
        } else {
            let value = self.buf[usize::from(self.uart_reader)];
            self.uart_reader += 1;
            Some(value)
        }
    }
}


```

接下来就是Uart本身的设计了。

## uartputc

这里用到了C侧的一个全局变量，我们将其引入bindgen_warpper.h

```c
extern volatile int panicked; // from printf.c
```

ffi后volatile的信息就没了，所以我们还需要：

```rust
#[inline]
fn is_panicked() -> bool {
      unsafe {addr_of!(panicked).read_volatile() != 0 }
}
```

实现过程中发现了一个坑--C侧如果panic了，就死循环
我本来想用panic!,但是多问了一句为啥C侧不用(C也有panic()函数)
原来是panic的默认处理会调用printf，进而调用到uart，所以在uart本身的错误处理不能这样

```rust
     if is_panicked() {
            loop {
                //这里不应该使用panic!
                //因为panic! 本身会调用printf，进而一路调用回来...
                //我们只能原地死循环
            }
        }
```

然后就是下面的实现，这里同时体现出了rust的好处和坏处...

```rust
        loop {
            {
                let mut buf = self.buffer.lock();
                if let Ok(_) = buf.push(c) {
                    //C代码的其他处理在push里封装好了
                    //锁会在离开putc后自动释放
                    self.start();
                    break;
                }
                else{
                    //理论上我们这里需要sleep，然后等待start函数来唤醒我们
                    //诶，问题来了...sleep需要CSpinLock(因为是C侧在进行操作)
                    //也就是说我们的RAII就失效了
                    //如果不传入锁，就会出现在sleep之前，锁释放之后的这一瞬间刚好被中断打断
                    //中断处理中尝试唤醒我们...但是我们还没睡
                    //就睡死了...对，死了
                    //所以我们要自己实现rust_sleep
                }
            }
        }
```

好处:
1.RAII自动处理了锁，我们不需要手动释放，出了作用域就自动释放了
2.我们的push和UartPointer进行了两重封装，这里非常简单
坏处:我们没办法用C版的sleep了，它是手动管理锁的

所以我们要实现rust_sleep.就把proctable.rs改名叫proc.rs，然后塞进去吧。

```rust
pub fn rust_sleep<T: ?Sized,U>(chan: &T, lock:SpinLockGuard<'_,U>){
    let p=unsafe{myproc()};
  
    unsafe{acquire(&mut ((*p).lock));}
    drop(lock);//释放锁
    unsafe{
    (*p).chan = chan as *const T as *mut c_void;
    (*p).state = procstate::SLEEPING;
    sched();
    (*p).chan=core::ptr::null_mut();
    release(&mut ((*p).lock));
    }
```

顺便做了对C版本的封装

```rust
/// 对 xv6 `sleep(chan, lk)` 的薄封装。
/// - 当前上下文允许睡眠，且调用前已经持有 `lock`。
pub  fn  sleep<T: ?Sized>(chan: &T, lock:&CSpinLock) {
    unsafe { c_sleep(chan
        as *const T
        as *const c_void
        as *mut c_void
        ,
         lock 
        as *const CSpinLock
        as *const spinlock
        as *mut spinlock) 
    };
    //需要进行一系列转换
    //这一连串转换好搞笑
}

/// 对 xv6 `wakeup(chan)` 的薄封装。
pub  fn wakeup<T: ?Sized>(chan: &T) {
    unsafe { c_wakeup(chan
        as *const T
        as *const c_void
        as *mut c_void
    ) };
}
}
```

这里我们可以和C版本对照一下，很有意思

```C
sleep(void *chan, struct spinlock *lk)
{
  struct proc *p = myproc();
  
  // Must acquire p->lock in order to
  // change p->state and then call sched.
  // Once we hold p->lock, we can be
  // guaranteed that we won't miss any wakeup
  // (wakeup locks p->lock),
  // so it's okay to release lk.

  acquire(&p->lock);  //DOC: sleeplock1
  release(lk);

  // Go to sleep.
  p->chan = chan;
  p->state = SLEEPING;

  sched();

  // Tidy up.
  p->chan = 0;

  // Reacquire original lock.
  release(&p->lock);
  acquire(lk);
}
```

终端看锁的处理。我们使用drop+RAII释放锁，这没问题。
问题在于，我们没有办法acquire锁，因为我们的Guard已经被释放了！
我是怎么处理的呢？将这个函数的行为定义为“需要用户额外获得锁”
这其实相当符合rust的所有权机制
你可以注意到，我们的lock是直接获得的所有权，而不是借用
因此，这个函数调用完毕后，外部就没有lock的所有权了
程序员想用lock(来访问数据)，就会发现lock没了，自然会去拿一个锁
而很显然，这个地方的锁早一点晚一点不会像开头那么影响

于是乎，我们只需要在else里调用

```rust
rust_sleep(& buf.uart_writer, buf);
```

即可。一轮loop结束后自然会再拿一个锁。
